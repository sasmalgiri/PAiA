Security + responsible disclosure

How PAiA is built to be secure, and how to report something if it isn't.

How PAiA is built

  • Local by default. The app's core loop runs entirely on your machine. Nothing leaves your device unless a specific opt-in feature is configured.
  • PII redaction pipeline. 11 categories of personally-identifiable information are scrubbed before any prompt leaves the renderer.
  • End-to-end encryption for sync. AES-256-GCM per object, PBKDF2-SHA-256 with 200,000 iterations from your passphrase. Storage operators see counts and sizes, not contents.
  • Ed25519 signatures. Licenses, beta invites, and trial extensions are signed and verified offline with an embedded public key.
  • Renderer isolation. Strict Content-Security-Policy. No inline scripts. Context isolation and a narrow contextBridge API. Sandboxed iframe for Canvas HTML/SVG.
  • Plugins run with full Node capabilities. Plugins are a trust surface; they're opt-in, scanned before load, and their declared contributes block is shown to the user for audit before enabling.
  • Signed installers on macOS + Windows (once certs are active — see our changelog for the first signed build date).
  • Permission model for OS-level enforcement. Every firewall / hosts / registry modification requires an admin prompt from the OS; PAiA never inherits elevation.

Reporting a vulnerability

Email security@paia.app. PGP available at paia.app/pgp.asc — please encrypt anything that includes a working exploit.

Please include:

  • PAiA version (Settings → About).
  • Your OS + architecture.
  • A clear description + reproduction steps + a proof of concept.
  • Your handle if you'd like credit in the release notes.

We will acknowledge within 3 business days and give you a progress update within 10 business days. Most fixes ship within 30 days for critical issues, 60 days otherwise.

In scope

  • The PAiA desktop application on any platform.
  • The license issuance and activation flow.
  • The Companion LAN HTTP server.
  • The local REST API server.
  • The classroom server + enforcement scripts.
  • The E2E-encrypted sync protocol.
  • The agent orchestrator and the Plugin SDK sandbox.

Out of scope

  • Vulnerabilities in third-party LLM providers (report to them).
  • Vulnerabilities in Electron / Chromium itself (report upstream first).
  • Self-XSS that requires the user to paste malicious content into their own chat input.
  • Denial of service via very large prompts, mic capture, or infinitely-looping agent goals.
  • Social engineering of our support staff.

Safe harbor

We will not pursue legal action against researchers who:

  • Make a good-faith effort to comply with this policy.
  • Avoid privacy violations, data destruction, and service interruption during testing.
  • Only interact with accounts they own or have explicit permission to test.
  • Give us reasonable time (90 days by default) to fix before public disclosure.

Rewards

We aren't running a public bug-bounty programme yet. For notable findings we may offer swag and — for the worst issues — a discretionary reward at USD 100–1000 depending on severity, novelty, and report quality. Ask before testing anything you'd expect a reward for.
